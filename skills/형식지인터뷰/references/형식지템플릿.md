# Phase 1 원업무 형식지 최종 산출물 템플릿

> 이 문서는 **AX 프로젝트의 일환으로 AX 매니저가 신청자의 원업무를 형식지로 추출**하기 위해 작성하는 기록지다.
> 작성 주체는 **AX 매니저 또는 AI 후처리**이며, 신청자가 직접 채우는 항목은 없다. 인터뷰·업무 시연·세션 로그·자료 검토 내용을 바탕으로 작성한다.
> 추출 깊이는 "아무것도 모르는 신청자의 후임자가 이 문서만 보고도 해당 업무를 수행할 수 있는 수준"을 지향한다.
> 구현 방식이나 자동화 방향은 적지 않고, 현재 사람이 실제로 일하는 방식을 기록한다.

## 1. 업무 정의: 5요소

**형식**: 5요소 체크

- 누가:
- 무엇을 대상으로:
- 누구를 위해:
- 어떤 주기로/언제:
- 어떤 맥락·목적으로:

**작성 힌트**:
- 각 항목을 1~2줄로 적어 5요소만으로 업무 정체가 드러나게 한다.
- "무엇을 대상으로"에서, 이 업무가 전체 대상 중 일부만 다룬다면 전체 규모와 실제로 처리하는 대상을 구분해 적는다. (어떤 기준으로 골라지는지는 5번 판단에서 다룬다.)

## 2. 시작 조건과 완료 조건

**형식**: 짧은 표

| 구분 | 조건/신호 | 관련 자료 또는 결과 | 비고 |
|---|---|---|---|
| 시작 조건 |  |  |  |
| 완료 조건 |  |  |  |

**작성 힌트**:
- 시작·완료 조건은 "언제 시작/끝났다고 인식하는가"이고, 입력·출력은 "무엇을 다루는가"다.
- 같은 메일이 시작 신호이면서 입력 자료일 수도 있다. 그 경우 두 칸 모두 적되 의미 축이 다름을 인지한다.
- 외부에서 검증 가능한 신호로 작성한다. "업무가 끝났다는 느낌"이 아니라 "팀장 회신 수신" 같은 확인 가능한 사건으로.

## 3. 입력 자료

**형식**: 표

| 입력 자료 | 형태/채널 | 출처 | 실제로 보는 부분 | 필수 여부 | 자주 빠지거나·변동되거나·달라지는 부분 | 사용 전 확인 사항 |
|---|---|---|---|---|---|---|
|  |  |  |  | 필수 / 조건부 / 참고 |  |  |

**형태/채널 칼럼 작성 시 참고할 예시 목록** (이 목록 중 하나로 한정하지 않는다 — 신청자가 실제 사용하는 형태를 그대로 적기): 슬랙, 이메일, 회의 내용, 세일즈맵, 공유 드라이브, 시스템 알림, 구두 요청, 문서, 표, 시스템 화면, 그 외 어떤 형태든.

**작성 힌트**:
- 비공식 채널(구두 보고·메신저·전화·회의 발언)도 빠짐없이 포함한다. 문서·시스템 자료만 적지 않는다.
- "실제로 보는 부분"은 자료 전체가 아니라 일부 필드·구간만 보는 경우를 명문화한다. 신청자가 실제로 어느 부분만 사용하는지를 그대로 기록한다.
- "자주 빠지거나·변동되거나·달라지는 부분"은 입력 자체의 변동 패턴이다. 흐름 전체의 예외는 8번에 적는다.
- 권한·정책·보안 때문에 자료 취득이 제약되면(예: 회사 계정에서 막혀 수동으로 받음) 해당 자료 옆에 `[권한·정책]` 태그를 붙인다. 실제로 부딪힌 제약만 기록하고 "구현하면 막힐 것" 같은 추정은 적지 않는다(Phase 2 영역). 태그된 항목은 11번에 종합된다.

## 4. 대표 사례 기반 업무 흐름

**형식**: 표 + 특이사항 메모

| 순서 | 단계명 | 실제로 한 일 | 이때 본 자료/화면 | 이 단계에서 확인한 것 | 단계 결과 | 판단/선택 여부 |
|---|---|---|---|---|---|---|
| 1 |  |  |  |  |  | 없음 / 있음 |
| 2 |  |  |  |  |  | 없음 / 있음 |
| 3 |  |  |  |  |  | 없음 / 있음 |

**본 사례의 특이사항** (해당 시 적기, 없으면 비워둠):
- 원래는 들어갔어야 하지만 본 사례에서는 빠진 단계:
- 원래는 빠져야 했지만 본 사례에서 포함된 단계:
- 본 사례에서만 일회적으로 발생한 변형·예외 처리:

**단계 쪼개기 기준**:
- 결과물이 한 단계 진척되는 지점
- 사용하는 도구·사람·장소·자료가 바뀌는 지점
- 판단이나 선택이 끼어드는 지점
- 시간대나 세션이 나뉘는 지점
- 다른 사람에게 인수인계한다면 자연스럽게 끊을 지점

**권장 단계 수**: 5~12개. 너무 촘촘한 클릭 단위가 아니라, 업무 단위로 쪼갠다.

**작성 힌트**:
- 이 표는 대표 사례를 출발점으로 삼되, 평소에도 반복되는 기본 흐름으로 정리한다. 대표 사례에만 해당하는 절차는 표 본문이 아니라 특이사항 메모에 분리한다.
- 출발점은 이상적인 절차가 아니라 최근 실제 사례 1건이다.
- "판단/선택 여부"가 `있음`인 단계는 5번에서 다시 다룬다 (단, 5번은 본 사례에 한정되지 않는다 — 5번 안내 참조).

## 5. 판단 지점과 판단 기준

**형식**: 카드.

본 업무 전체에서 발생하는 판단 지점을 모두 수집한다. **4번 대표 사례에 한정하지 않는다** — 4번 사례에서 발견된 판단 + 사례에 나오지 않은 판단(다른 사례·일반 패턴·신청자 설명·세션 로그)을 함께 적는다. 신청자가 명시적으로 말한 판단뿐 아니라, 세션 로그·녹취·시연을 보고 발견한 판단도 포함한다.

```markdown
### 판단 지점 1
- 발생 상황:
- 실제로 고민한 질문:
- 선택한 결과:
- 참고한 정보:
- 이렇게 판단한 기준:
- 다르게 처리되는 경우:
```

```markdown
### 판단 지점 2
- 발생 상황:
- 실제로 고민한 질문:
- 선택한 결과:
- 참고한 정보:
- 이렇게 판단한 기준:
- 다르게 처리되는 경우:
```

**작성 힌트**:
- 판단 1건이 깊은 서술이 될 수 있으므로 카드 형식으로 호흡 있게 적는다.
- 포함/제외/보류, 우선순위, 수정 필요 여부, 추가 확인 여부, 전달 대상 결정 등이 모두 판단에 해당할 수 있다.
- 판단이 10건을 넘어가면 표로 압축해도 된다.
- 4번 사례에서 발견한 판단인지 다른 출처인지를 "발생 상황"에 짧게 표시해두면 추적이 쉽다 (예: "발생 상황: 4번 단계 3에서 …" 또는 "발생 상황: 신청자 일반 설명 — …").
- **신청자 직접 입력 판단값**: 신청자가 점수·등급·가능성·상태 같은 걸 직접 매겨 시스템·시트에 입력하는 값도 판단의 일종이다. 여기 기록하되 "선택한 결과"에 그 값이 시스템에 기록됨을 표시한다. 그 값이 흐름의 한 단계이거나 다음 업무의 시작 신호이기도 하면 3번·2번에서 함께 표시한다.
- **전체 중 일부만 다루는 경우**: 이 업무가 전체 대상 중 일부만 처리한다면, 그게 (a) 본인이 매번 적극적으로 선별하는 판단인지, (b) 적극적 판단 없이 원래 그 조건만 처리·접수하는 것인지 구분해 적는다. (b)라면 그 조건이 사업부 원칙인지·업무 정의인지 출처도 함께 적는다. 전체 규모와 처리 대상이 정해지는 기준은 1·2·3·5번 중 자연스러운 곳에 적으면 된다 — 위치보다 내용이 누락되지 않는 게 핵심.

## 6. 산출물과 완료 기준

**형식**: 구조화된 줄글

- 산출되는 것:
- 산출물 형태:
  - 참고 예시 (이 중 하나로 한정 X — 신청자 산출물의 실제 형태를 그대로 적기): 문서, 표, 메일, 메시지, 시스템 기록, 노트, 구두 공유, 그 외 어떤 형태든
- 템플릿·양식·규칙 유무:
- 필수 구성요소:
- 저장 또는 기록 위치:
- 전달 대상:
- 전달 방식:
- 완료로 인정되는 상태:

**작성 힌트**:
- 산출물이 파일 형태라는 보장은 없다. 세일즈맵 노트, 슬랙 메시지, 시스템 기록, 구두 공유 등도 산출물로 본다.
- "저장 또는 기록 위치"는 팀 공유 위치를 우선 적는다. 개인 PC 경로가 있으면 그대로 적되 후속 정리 대상으로 인지한다.
- 산출물 저장·전달이 권한·정책·보안으로 제약되면(예: 공유 드라이브 접근 권한 없어 메일로 전달) 해당 항목 옆에 `[권한·정책]` 태그를 붙인다. 태그된 항목은 11번에 종합된다.

## 7. 좋은 결과물 조건

**형식**: 카드. 조건이 많으면 표로 바꿔도 된다.

```markdown
### 조건 1
- 충족해야 하는 조건:
- 통과 기준/수준:
- 기준 출처: 매뉴얼 / 관행 / 수신자 피드백 / 본인 판단 / 없음
- 확인 방식: 누가, 언제, 어떻게 확인하는가
- 미충족 시 처리: 재작업 / 통과 / 저장만 / 다음 사이클 이월 / 기타
```

```markdown
### 조건 2
- 충족해야 하는 조건:
- 통과 기준/수준:
- 기준 출처: 매뉴얼 / 관행 / 수신자 피드백 / 본인 판단 / 없음
- 확인 방식: 누가, 언제, 어떻게 확인하는가
- 미충족 시 처리: 재작업 / 통과 / 저장만 / 다음 사이클 이월 / 기타
```

**작성 힌트**:
- "충족해야 하는 조건"은 *무엇*이고, "통과 기준/수준"은 *어느 정도*면 통과인지다. 예: 조건 = "오타가 없어야 한다" / 통과 기준 = "0개여야 통과" vs "의미 전달에 문제 없으면 acceptable".
- 두 차원은 다르다. 이 구분이 없으면 재작업 판단이 모호해진다.
- 명시 기준이 없고 관행·피드백·본인 판단으로만 운영되는 경우도 그대로 적는다.

## 8. 주의할 점·실수·예외·애매한 상황

**형식**: 줄글 4섹션 + 빈 경우 라벨

**사용 가능한 라벨** (빈 칸을 "없음"으로 오해하지 않게 표시):
- `[확인된 사례 없음]` — 이번 세션에서 실제로 사례가 발견되지 않음
- `[신청자 응답 어려움]` — 답하기 어려움 (10번 미확인·보완으로 이동 권장)
- `[추가 확인 필요]` — 후속 발굴 대상

**라벨 부착 위치**: 각 섹션(`### 주의할 점` 등) 안에 둔다. 섹션 전체가 비면 라벨만 단독 행으로, 일부 사례만 있고 나머지가 비면 사례 행 뒤 라벨 행 추가.

```markdown
### 주의할 점
- (사례 1) 상황: ... / 왜 중요: ... / 처리: ...
- (사례 2) 상황: ... / 왜 중요: ... / 처리: ...

### 자주 생기는 실수
[확인된 사례 없음]

### 예외 상황
- (사례 1) 상황: ... / 처리: ...
- [추가 확인 필요]

### 애매한 상황
[신청자 응답 어려움]
```

**작성 힌트**:
- 신청자가 "다 설명했다"고 해도 예외·실수는 자주 인지되지 않은 채 흘러간다. 세션 로그·시연 영상에서 발견할 수 있다.
- 예외는 "예외를 다 떠올려라"가 아니라 "정상 흐름대로 안 되는 상황"으로 묻는다. 입력 누락, 기준 충돌, 담당자 부재, 자료 형식 다름, 수신자 반려 등.
- 빈 칸에는 반드시 라벨을 남긴다. "없음"으로 추론되어 후속 단계에서 잘못 안심하지 않게 한다.

## 9. 형식지 작성 근거

**형식**: 표

| 근거 유형 | 자료 또는 사례 위치 | 영향 항목 | (선택) 확인한 내용 |
|---|---|---|---|
|  |  |  |  |

**근거 유형 예시** (한정 X — 실제 출처를 적기): 실제 입력 자료, 실제 산출물, 화면 공유, 인터뷰 메모, 세션 로그, 신청자 설명, 기타.

**작성 힌트**:
- 본 항목은 AX 매니저가 본 문서의 신뢰성을 스스로 검증하거나, 후속 인터뷰·Phase 2 인계 시 출처를 다시 추적할 수 있도록 남기는 기록이다.
- "영향 항목"은 어느 본문 섹션(1~11)에 반영됐는지를 적는다. 없으면 비워도 된다.
- "확인한 내용"은 선택 칼럼이다. 근거에서 *무엇을* 확인했는지를 강조하고 싶을 때만 채운다. 기본은 비워둔다 — 섹션이 무거워지지 않도록.

## 10. 미확인·보완 필요 항목

**형식**: 표

| 항목 | 유형 | 왜 남았는가 | 보완 방법 | 상태 |
|---|---|---|---|---|
|  |  |  |  | 미확인 / 보완 중 / 보완 완료 |

**유형 5종**: 자료 추가 확인 / 기준 표현 미완성 / 신청자 확인 필요 / 근거 부족 / 암묵지·직감 영역

**작성 힌트**:
- `암묵지·직감 영역`은 LLM 후처리까지 했는데도 기준화되지 않은 잔여 영역에만 쓴다. 기본값이 아니라 마지막 안전망이다.
- 후속 인터뷰·자료 확보로 채울 수 있는 항목은 다른 4종으로 분류한다.
- 비어 있는 게 정상인 경우(이번 세션에서 충분히 풀린 경우)는 표 전체를 비워두지 말고 한 줄에 `[현재까지 보완 필요 항목 없음]` 라벨을 남긴다.

## 11. 보충 메모 및 후속 참고

본 섹션은 1~10번 본문에 담기 어려운 작업 메모와 다음 단계 참고 사항을 적는 자리다. 본 문서 전체가 AX 매니저용 기록이므로 별도 "내부"라는 의미보다는 본문 외 보충 정보로 본다.

- 신청자에게 직접 확인한 내용:
- AX 매니저가 세션 로그·자료를 보고 해석한 내용 (신청자 검증 전):
- 아직 확정하면 안 되는 내용:
- 다음 단계(후속 확인·Phase 2 인계)에서 참고할 원업무상 특징:
- 구조적 제약(권한·정책·보안·PII) 종합: 본문 3·4·6번에 `[권한·정책]` 태그된 항목을 한곳에 모은다. 없으면 비워둔다.

**작성 힌트**:
- "확인한 내용"과 "해석한 내용"을 구분해 적는다. 후자는 신청자 검증 전이므로 후속 단계에서 재확인 필요.
- 1~10번에 들어가지 못한 정보·관찰·의문점도 여기에 자유롭게 적는다.

## 작성 시 주의어

아래 표현이 본문에 등장하면, **신청자 발화나 사실 기록이 아니라 AX 매니저의 구현 판단으로 흘렀는지** 확인한다. 단어 자체를 금지하지는 않는다 — 현장 발화·사실 기록이면 그대로 남긴다.

- 자동화 / 구현 / 자동화 가능성 / 자동화 난이도
- LLM / API / MCP / DB / skill / 프롬프트 / 에이전트 / Few-shot
- 효율화 / 최적화 / 개선 방향 / 개선 여지 / 적합성 / 간소화
- "이 업무는 ~유형이므로", "이 업무 유형은"
- "간단하다", "단순하다", "복잡하다"
- "반드시 decision tree로", "decision tree 필수"
