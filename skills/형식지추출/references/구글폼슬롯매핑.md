# 구글폼 ↔ 형식지 11칸 매핑 (references)

> [전] 단계가 이 파일을 읽어 "구글폼이 어느 칸을 미리 채웠나 / 어디를 인터뷰에서 파야 하나 / 어디에 보강 질문을 붙이나"를 판단한다.
> ※ draft. 실제 플러그인에선 `skills/형식지인터뷰/references/구글폼슬롯매핑.md`.

## 0. 사용 원칙 (요약)

1. 구글폼 응답을 11칸에 **풀(pool)** — 한 질문이 여러 칸을 비추므로 *전부* 모아서 본다.
2. 칸마다 채움 정도를 **충분 / 부족 / 빔** 3단으로 본다.
3. **빔·부족 칸 + "구글폼이 못 주는 깊이" 칸 → 인터뷰 질문에 반드시 포함.**
4. 보강 질문은 **(a) 구글폼 신호·근거가 있거나 (b) 채워졌지만 부족할 때**만. *지어내기 금지·적게·열린 형태*.

## 4. ★ 구글폼이 끝내 못 주는 깊이 (= 인터뷰가 무조건 파야 함)

질문을 다 모아도(many-to-many 풀해도) 남는 칸. **[전]은 이 칸들의 표준 질문을 항상 인터뷰질문에 넣는다. 구글폼에 없는 정보(분야·범위 등)는 단정 말고 *물어라* — 지어내기 금지.**

- **1번** 수혜자 · 맥락/목적 · **전체/일부 담당범위**(구글폼이 분야·범위 미언급 시)
- **2번** 시작/완료 조건
- **5번** 판단 *기준* (깊이)
- **7번** 통과 *기준* (깊이)
- **8번** 실수·예외·애매 (깊이)

## 5. 보강 질문 — 신호별 앵커 (추가 질문)

**조건**: 아래 *신호가 있거나*, 또는 *해당 칸이 부족하게 채워졌을* 때만. *지어내기 금지·적게·열린 형태("X 하시죠?" 금지).*

| 구글폼 신호 | 보강 질문 (예) | 들어갈 칸 |
|---|---|---|
| Q9에 **사내DB·외부API/MCP** | "그 시스템 접근에 권한·보안 제약 있어요?" | 3 `[권한·정책]` |
| Q9에 **설문·피드백** | "그 응답 그대로 쓰세요, 거르세요? 거르는 기준은?" | 5 |
| Q7 = **아니오/일부** | "결과물이 *뭐에 따라* 달라지나요?" | 5 |
| Q10 = **배포·운영 중** | "지금 그 도구, 어디까지 자동이고 어디부터 손이 가요?" | 11(+8) |
| Q11 페인 = "OO이 번거롭다" | "그 OO, *구체적으로 어느 단계에서 왜* 번거로워요?" | 8(+5) |
| **Q8·Q12·Q13 첨부 있음** | (세션 전 그 양식 보고) "이 칸은 어떻게 채우세요? 비는 칸 있어요?" | 3·5·6 |

**렌즈 (추가 질문 아님)**: Q1 팀 → 그 도메인 어휘로 전체 질문을 재구성 (자료→교안 등).

## 6. 가드레일

- 보강 질문도 **열린 형태** — 추측으로 구체 질문 만들어 신청자 끌고 가지 않기.
- 첨부(Q8·12·13)를 못 읽으면 "이 자료는 직접 확인 필요" 안내 (무시 X).
- 신청자 행은 **이메일(Q3) 매칭** — 이름 중복 방지.
- 세션 폴더엔 *해당 신청자* 행만 남김 (전체 시트 X) — 개인정보.
